Workspace Overview

IronKey is a Cargo workspace (resolver = "2") under a single Cargo.toml. All crates share common dependency versions via [workspace.dependencies]. The workspace compiles with opt-level = 3, LTO, single codegen unit, and strip = true for release builds — producing a compact, fast binary suitable for a read-only USB image. 

ironkey (workspace)
├── ironkey-app          ← binary entry point
│   └── depends on: ironkey-ui
├── ironkey-ui           ← iced 0.14 PaneGrid application
│   ├── depends on: ironkey-drives
│   ├── depends on: ironkey-mount
│   ├── depends on: ironkey-browser
│   └── depends on: ironkey-terminal
├── ironkey-drives       ← /sys/block/ enumeration + SMART + disk ops
├── ironkey-mount        ← filesystem detection + mount/unmount
├── ironkey-browser      ← file listing, ops, search, viewer
├── ironkey-terminal     ← PTY + VTE ANSI renderer
└── xtask                ← cargo xtask build helpers

ironkey-app

Binary crate. src/main.rs initialises env_logger (default level info) and calls ironkey_ui::run(). If the app returns an error it logs it and exits with code 1.

src/main.rs 
fn main() {
    env_logger::Builder::from_env(
        env_logger::Env::default().default_filter_or("info"),
    ).init();
    log::info!("IronKey v{} starting", env!("CARGO_PKG_VERSION"));
    if let Err(e) = ironkey_ui::run() {
        log::error!("Application error: {:?}", e);
        std::process::exit(1);
    }
}

Dependencies: anyhow, log, env_logger, ironkey-ui

ironkey-ui

Library crate. Implements the iced application entry point and owns all application state. The entire IronKey UI is one iced application with a three-pane layout.

Key Components

ComponentRole
IronKeyAppTop-level application state struct
MessageExhaustive enum of all user and system events (drive actions, browser ops, terminal input, modal events, system tick, I/O stats)
PaneGridThree resizable panes identified by PaneKind: Drives, Browser, Terminal
2-second tickiced::time::every for drive enumeration refresh and I/O stats polling
ModalOption<Modal> enum: Format, Passphrase, Confirm, Progress, Info, FileViewer, Properties, Shutdown, Reboot, NewTable

Module Structure

app · panels (drives / browser / terminal) · modals (one file per modal variant) · theme · widgets · io_stats

Window config: The window is configured 1280×800 for desktop development builds (fullscreen: false). On hardware the IronKey ISO sets fullscreen: true via a compile-time feature flag.

Dependencies: iced 0.14 (features: image, tokio), sysinfo, chrono, all other ironkey crates

ironkey-drives

Library crate. Three modules handle all drive-related operations.

Module: enumerate

Parses /sys/block/ to build Vec<DriveInfo>. Each DriveInfo contains model, size, removable flag, and Vec<PartitionInfo>. Reads /proc/self/mountinfo to determine PartitionStatus (mounted / unmounted / busy).

Module: smart

Issues ATA IDENTIFY DEVICE and SMART READ DATA commands via nix::ioctl_readwrite! macros. Returns Option<SmartData> containing: overall health pass/fail, temperature, power-on hours, reallocated sector count, pending sector count, uncorrectable sector count, read error rate.

Module: ops

Blocking disk operations intended to run inside tokio::task::spawn_blocking:

  • format_partition(opts: FormatOpts) — runs the appropriate mkfs.* binary
  • clone_partition(src, dst, tx: Sender<CloneProgress>) — block-level copy with live progress
  • wipe_partition(dev, mode: WipeMode) — fills with zeros (Fast) or random data (Secure)

Key Public Types

DriveInfo · PartitionInfo · DriveStatus · PartitionStatus · SmartData · FormatOpts · FormatFsType · WipeMode · CloneProgress

Dependencies: nix (features: ioctl, fs, mount) — Linux-only

ironkey-mount

Library crate. Detects filesystem type by reading magic bytes, then dispatches the appropriate mount strategy.

ModuleStrategy
detect Reads first 512 bytes, matches magic signatures: LUKS (LUKS\xba\xbe), NTFS (NTFS ), APFS, BitLocker, ext2/3/4, btrfs, XFS, F2FS, vfat
luks Runs cryptsetup luksOpen <dev> ironkey_<uuid> then mounts the mapper device
ntfs Tries kernel ntfs3 module first, falls back to ntfs-3g FUSE
apfs Uses apfs-fuse FUSE driver (read-only; write support not available)
bitlocker Uses dislocker to expose a virtual device, then mounts the inner NTFS
generic Calls mount(2) via nix with the detected fstype string

ironkey-browser

Library crate. Four modules cover all file browser functionality.

Module: listing

list_directory(path: &Path) -> Result<Vec<FileEntry>>. Each FileEntry has: name, FileKind (Dir/File/Symlink), size in bytes, modified timestamp, Unix permissions.

Module: ops

File operations, all returning anyhow::Result<()>:

  • copy_file(src, dst) — recursive for directories
  • move_file(src, dst) — rename, falls back to copy+delete across devices
  • rename_file(path, new_name)
  • delete_file(path) — recursive delete

Module: search

search_directory(root: &Path, opts: SearchOpts) -> Result<Vec<SearchResult>>. Uses walkdir for recursive traversal. Filters by filename glob, content substring (text files only), max depth. Returns path, file kind, size, and match context.

Module: viewer

view_file(path: &Path) -> Result<ViewedContent>. Detects MIME type via magic bytes + extension, then returns one of:

  • ViewedContent::Text(String) — UTF-8 content (max ~1 MB)
  • ViewedContent::Image(Vec<u8>) — raw bytes for iced's image widget
  • ViewedContent::Hex(Vec<u8>) — first 4 KB for hex display
  • ViewedContent::TooLarge — file exceeds viewer limit

ironkey-terminal

Library crate. PTY functionality is Linux-only via portable-pty.

TerminalState

Screen buffer as Vec<Vec<Cell>>, each Cell holding a character and SGR attributes (fg/bg colour, bold, italic, underline). Also tracks cursor position and scrollback buffer.

VTE Integration

vte::Parser feeds raw bytes from the PTY into TerminalState. The parser handles: cursor movement (CSI A/B/C/D), erase sequences (ED, EL), SGR colour codes (8-colour, 256-colour, and truecolor), and alternate screen buffer.

PTY Spawner

Uses portable-pty::native_pty_system() to create a pty pair sized to the panel dimensions. Spawns zsh with the IronKey environment variables. The master fd is polled via an iced Subscription.

Dependencies: vte 0.15, portable-pty 0.9 (Linux only), thiserror, log

xtask

Binary crate providing cargo xtask <command> build helpers. Not included in the USB image. Used during development for tasks like version bumping and running pre-release checks.

Tip: Run cargo xtask --help to see available build automation commands.

OS Layer (os/)

The os/ directory contains everything needed to produce a bootable USB image. The os/build.sh script orchestrates 7 phases to produce a hybrid BIOS+UEFI bootable ISO:

  1. Compile — cargo build -p ironkey-app --release with LTO + strip. The resulting binary is ~10–20 MB.
  2. Bootstrap — debootstrap --variant=minbase bookworm creates a ~300 MB minimal Debian rootfs containing systemd, cage (Wayland kiosk compositor), zsh, and udev.
  3. Configure — Copies the binary to /usr/bin/ironkey, installs shell configs (/etc/zsh/), starship config, and enables ironkey.service via a symlink in multi-user.target.wants/.
  4. squashfs — mksquashfs -comp zstd -Xcompression-level 19 compresses the rootfs to ~150–200 MB. /proc, /sys, and /dev are excluded.
  5. Kernel — Extracts vmlinuz and initramfs.img from the debootstrap rootfs.
  6. GRUB config — Copies os/grub/grub.cfg which sets timeout=0, loglevel=0, and passes quiet splash=0 to the kernel for a silent boot.
  7. Hybrid ISO — grub-mkrescue produces an ISO image with both a BIOS MBR and a UEFI ESP, bootable via dd or Rufus.

Runtime Behaviour

  • GRUB → Linux kernel → systemd init
  • systemd starts ironkey.service on multi-user.target
  • ironkey.service runs cage -- ironkey on tty1 as root
  • cage is a Wayland kiosk compositor — IronKey is the sole Wayland client
  • All writes go to tmpfs mounts: /tmp (256 MB), /var (128 MB), /run (64 MB)
  • The squashfs rootfs is read-only; the live USB is fully ephemeral
Shell config: The os/rootfs/etc/zsh/ directory ships zshrc and ironkey-aliases.zsh. These are copied into the rootfs during the Configure phase and provide all the Rust-native CLI aliases and ik-mount helper function.

Build Command

Build the ISO 
sudo bash os/build.sh ironkey-$(git describe --tags).iso
Root required: os/build.sh must run as root because debootstrap, mksquashfs, and grub-mkrescue require root privileges and access to loop devices.
